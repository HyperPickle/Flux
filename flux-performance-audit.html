Flux — Performance & Accuracy Audit

A code-level deep dive into process-energy accuracy and the app’s own resource footprint.
Scope: Sources/Flux/*.swift  ·  1,876 LOC  ·  Audit date 2026-06-20  ·  branch main

Two questions were investigated:

1. Why does Flux under-report processes that macOS flags as high-energy?
Because Flux and macOS measure different things over different time windows. Flux ranks processes by top’s instantaneous 5-second POWER column; macOS Battery settings ranks by a sustained, coalition-aggregated energy average over hours. On top of that, a name-matching bug makes whole categories of heavy work — most notably every Safari / WebKit tab — invisible to Flux.

2. Why does Flux sometimes rank itself as medium impact?
Flux continuously runs a child top process every 5 s and parses all ~970 process lines on this machine, hopping to the main actor several times per line. That CPU cost — plus the child top’s own cost, which macOS rolls into Flux’s coalition — is real and self-inflicted. The good news: it is almost entirely fixable without changing what the app shows.

§0How Flux measures energy today

All process data comes from a single long-lived child process spawned in BatteryMonitor.startTopStream():

// BatteryMonitor.swift:258–260
process.executableURL = URL(fileURLWithPath: "/usr/bin/top")
process.arguments = ["-l", "0", "-s", "5",
                     "-stats", "pid,command,cpu,power,mem", "-o", "cpu"]

Every 5 seconds top emits a full frame; parseTopLine() reads it line-by-line, attributes each process row to a running application by name substring matching, sums POWER/CPU/MEM per app, and the top 8 become topEnergyApps. The colored "Low/Medium/High/Critical" badge is derived in HeatLevel.init(power:cpu:) from that POWER value (falling back to CPU when power is 0).

The core conceptual gap. top’s POWER column is an instantaneous energy-impact score for the last sample interval. The macOS Battery pane’s “Apps Using Significant Energy” is a sustained average over a long window (since last charge / multi-hour), aggregated across each app’s entire process coalition (parent + all helpers + XPC children). Flux compares a snapshot against macOS’s long exposure — they will routinely disagree even when both are working correctly.

§1Accuracy of high-usage process monitoring

Critical wrong/missing data High materially misleading Medium noticeable skew Low minor

A1Safari & all WebKit content is completely invisibleCritical

This is the single most likely explanation for “macOS shows process X as significant but Flux shows it low.”

Safari does its real work in helper processes named com.apple.WebKit.WebContent, com.apple.WebKit.GPU, com.apple.WebKit.Networking. None of these strings contain “Safari”, so the app-matching loop never attributes them to Safari:

// BatteryMonitor.swift:337–342  — substring match against app names
for appName in appNames {
    if commandName.localizedCaseInsensitiveContains(appName) ||
       appName.localizedCaseInsensitiveContains(commandName...) {
            matchedApp = appName; break
    }
}

With no match, the row falls through to the background-process branch — which then explicitly drops everything Apple-owned:

// BatteryMonitor.swift:349–351
let blocked = backgroundProcessBlockList.contains(lower)
    || lower.hasPrefix("com.apple.")   // ← kills WebKit content procs
    || lower.hasPrefix("com.google.")

Net effect: a Safari tab pegging a CPU core contributes zero to Flux’s Safari row and is also hidden from the background list. macOS, which aggregates the WebKit coalition under Safari, correctly shows it as significant.

Asymmetry worth noting: Chrome works because its helpers are literally named Google Chrome Helper (Renderer) — that string contains “Google Chrome”, so it matches. Electron apps (Slack, VS Code → Code Helper, Discord) also match. Safari/WebKit is the conspicuous failure, and it is the system browser, so it hits a large fraction of users.

Fix direction: attribute by process coalition or parent PID rather than command-name substrings. The reliable primitive is proc_pid_coalitionid() / the proc_listcoalitions family, or mapping responsible-PID via responsibility_get_pid_responsible_for_pid(). A cheap stopgap: special-case com.apple.WebKit.* → “Safari” and stop blanket-blocking com.apple.* when it carries real activity.

A2Instantaneous snapshot vs. macOS’s sustained averageCritical

Flux replaces topEnergyApps wholesale every 5 seconds with the current interval’s numbers. A process that did heavy work 90 seconds ago but is idle now reads Low in Flux, yet macOS still lists it as a significant energy user because its multi-hour average is high. The reverse also happens (a brief spike during your glance looks alarming).

The on-card sparkline does retain history (appHistory, up to 720 points), but the ranking, the Heat badge, and the Drain pill all use only the latest instantaneous sample — so the headline classification has no memory.

Why this matters for your question: even with perfect per-process data, a 5-second snapshot and an 8-hour average are different statistics. To match user intuition (“what’s draining my battery”), rank by a trailing average (e.g. EMA over the last 1–5 min, which you already compute for the sparkline) instead of the last sample.
A3The first top frame is cumulative-since-launch, not a deltaHigh

In logging mode (-l 0), top’s first sample reports CPU/POWER accumulated since each process started; only subsequent frames are per-interval deltas. Flux parses the first frame identically to the rest (BatteryMonitor.swift:315–357), so for roughly the first 5 seconds after launch the energy ranking is dominated by whatever has run longest since boot — not what is busy now.

Fix: discard the first complete frame before populating the UI (skip until the second PID header is seen).

A4Substring matching produces silent mis-attribution & collisionsHigh

The match is bidirectional contains with a break on first hit, iterating apps in NSWorkspace order:

  • Short names collide: an app named “Code”, “Notes”, or “1” can swallow unrelated rows whose command merely contains the substring.
  • First-match-wins is order-dependent: if two running apps both match a row, the one that happens to come first in the workspace list claims the energy — non-deterministic across launches.
  • Truncation: top clips the COMMAND column to a fixed width; long bundle names get cut, breaking the substring test intermittently.

Fix: match on PID → NSRunningApplication(processIdentifier:) (exact, O(1)) instead of fuzzy strings. You already have the PID in parts[0] and currently throw it away.

A5CPU is normalized to whole-machine percent, making single-thread hogs look smallMedium

BatteryMonitor.swift:331–332 divides raw CPU by core count: cpu = rawCpu / cores. A single-threaded process maxing one core on this 10-core-class machine shows as ~10%, where Activity Monitor would show ~100%. Combined with the HeatLevel CPU fallback thresholds (Low <10%, Medium <30%), a fully pegged single core can register as merely “Medium” — again reading lower than the user expects from Activity Monitor.

This is a defensible design choice (percent of total machine capacity), but it is a second, independent reason Flux numbers look low next to Apple’s tools. Consider showing the Activity-Monitor-style per-core figure, or labeling the unit explicitly.

A6POWER is summed but treated as a unitless score; thresholds are arbitraryMedium

totalEnergyImpact sums top’s POWER across matched apps and feeds DrainLevel (thresholds 20 / 60 / 150) and HeatLevel (5 / 20 / 50). These constants don’t correspond to watts or any calibrated unit — they’re tuned to top’s internal score, which also varies by Mac model and macOS version. The real battery draw is already measured accurately elsewhere (fetchBatteryWatts() reads true amperage×voltage from AppleSmartBattery); the per-app POWER score is not commensurate with it. Worth cross-checking the badge thresholds against actual wattage, or at least documenting that “Drain” is a relative index.

A7Background-process activity filter can hide moderate-but-steady drainsLow

When “Include Background Processes” is on, a row is only kept if power > 0 || rawCpu > 3.0 (BatteryMonitor.swift:352). A daemon sitting at ~2% CPU continuously (a meaningful battery cost over hours) never appears, while it would show in macOS’s averaged view. Low severity because it’s opt-in, but it compounds A2.

Part 1 verdict. Flux’s readings aren’t “wrong” so much as measuring a different quantity than the Battery pane, with two genuine bugs layered on top (A1 WebKit blind spot, A3 first-frame). Fixing A1 + A4 (PID/coalition attribution) and A2 (rank by trailing average) would bring Flux into broad agreement with macOS for the cases you noticed.

§2Flux’s own performance & energy footprint

Why Flux occasionally shows up as a Medium impact process — in priority order of contribution.

B1The child top process is the headline cost — and macOS bills it to FluxHigh

Flux keeps a top -l 0 -s 5 running for the app’s entire lifetime. Every 5 seconds top walks the full process table (972 processes on this machine right now), computes per-process energy, and prints it. That scan is not free — it is exactly the kind of periodic full-table work that registers as measurable CPU.

The key insight for your question: top is a child of Flux, so it shares Flux’s process coalition. macOS’s energy accounting (the same coalition rollup discussed in A1) attributes the child’s CPU/energy back to Flux. So when you see “Flux — Medium,” a large part of that number is actually top doing the measuring. The observer is changing the measurement.

Fix direction: replace the external top with an in-process sampler — host_processor_info() for system CPU, proc_pid_rusage(... RUSAGE_INFO_V*) / proc_pidinfo for per-process CPU and energy. This eliminates the child process entirely, removes the fork/parse overhead, and gives you real per-PID energy counters (which also fixes A1/A4). It is the highest-leverage change in this audit.

B2~3 main-actor hops per process line, every 5 s (~2,900 hops/frame)High

parseTopLine is nonisolated, but for each data row it bounces to the main actor several times:

// per process row — BatteryMonitor.swift:323–344
} else if await getParsing() {          // MainActor.run #1
    ...
    let cores = await getCoreCount()    // MainActor.run #2  (a constant!)
    ...
    await addToSample(...)              // MainActor.run #3
}

At 972 processes that is on the order of ~2,900 main-actor context switches every 5 seconds, all competing with SwiftUI on the main thread. Two of the three are gratuitous:

  • getCoreCount() hops to the main actor to read an immutable constant — it never needs to.
  • getParsing() reads a flag that the same parse loop owns; it could be a local in a non-actor parser.

Fix: accumulate the whole frame in local (non-isolated) variables and hop to the main actor once per frame to publish. That alone cuts main-thread traffic by ~99%.

B3Byte-by-byte async line reading adds per-element overheadMedium

BatteryMonitor.swift:269 consumes fileHandle.bytes.lines. FileHandle.AsyncBytes buffers its syscalls, but it still vends and the .lines wrapper still assembles strings through the async-sequence machinery one element at a time. For a multi-kilobyte frame of ~970 lines every 5 s, that’s avoidable awaiting overhead. Reading fixed chunks and splitting on \n yourself is markedly cheaper. (Mostly moot once B1 removes top altogether.)

B4Per-render EMA recomputation across 8 cardsMedium

AppCard.smoothedHistory is a computed property, so SwiftUI re-runs the full exponential-moving-average over up to 720 points on every render of every card, and an expanded card renders two Charts off it. Because topEnergyApps is replaced every 5 s, all cards re-render and re-smooth on every frame. Memoize the smoothed series (compute once when the data point is appended in finalizeUIUpdate, store it on the model) instead of recomputing in body.

B55-second wake cadence prevents deep idleMedium

Independent of how much work each sample does, waking every 5 s keeps the CPU out of its lowest-power idle states. For a passive menu-bar meter, 5 s is aggressive — the panel is usually closed. Consider: sample at 5 s only while the popover is open; back off to 15–30 s (or pause) when it’s closed. The MenuBarExtra window’s appear/disappear is a natural trigger. This is the biggest single battery win after B1.

B6Unbounded appHistory dictionary growthLow

appHistory caps points per app at 720 but never evicts apps that have stopped running (BatteryMonitor.swift:67,387–391). Over a long uptime with many transient processes (especially if A1/background mode is broadened), the key set grows monotonically. Minor today; evict entries not seen for N frames.

B7Debug logging to /tmp and print() left in shipping pathsLow

The Launch-at-Login toggle writes [DEBUG-lal] lines to /tmp/flux_lal.txt on every change (SettingsView.swift:129–139), and BatteryMonitor.init emits several print() calls. Negligible for battery, but it’s debug residue that should be removed or gated behind #if DEBUG before release.

Part 2 verdict. The Medium self-rating is real and explainable: a child top whose cost coalitions back into Flux (B1), amplified by thousands of needless main-actor hops (B2) and a 5 s always-on cadence (B5). Moving to in-process sampling and frame-batched publishing should drop Flux to consistently Low.
